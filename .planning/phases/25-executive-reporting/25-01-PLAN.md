Add an ExecutiveReport model to schema.prisma mirroring the Report model shape per D-01. Fields:

- id (String @id @default(cuid()))
- clientId (String, FK to User.id, onDelete: Cascade) -- scoped to client, not assessment, because executive reports span multiple assessments
- advisorProfileId (String, FK to AdvisorProfile.id, onDelete: Cascade)
- version (Int) -- monotonic per (clientId, advisorProfileId)
- status (ReportStatus, @default(DRAFT)) -- reuse existing ReportStatus enum (DRAFT/PUBLISHED/SUPERSEDED)
- reportingPeriodStart (DateTime)
- reportingPeriodEnd (DateTime)
- executiveSnapshotData (Json?) -- null on DRAFT rows, populated at publish from buildExecutiveReportSnapshot
- brandingSnapshot (Json?) -- frozen branding at publish time
- advisorNotes (String?) -- editable on DRAFT, frozen on publish (per D-18)
- meetingAgenda (String?) -- editable on DRAFT, frozen on publish (per D-18)
- discussionPrompts (Json?) -- String[] stored as Json, frozen on publish (per D-18)
- publishedAt (DateTime?)
- publishedById (String?, FK to User.id, onDelete: SetNull)
- createdAt (DateTime @default(now()))
- updatedAt (DateTime @updatedAt)

Add relations: client -> User, advisorProfile -> AdvisorProfile, publishedBy -> User. Add the reverse relation arrays on User and AdvisorProfile models.

Indexes:
- Unique on (clientId, advisorProfileId, version)
- Composite on (clientId, advisorProfileId, status)
- Partial unique index enforcing one DRAFT per (clientId, advisorProfileId) -- this requires raw SQL in the migration since Prisma DSL does not support partial unique indexes

Create the migration with `npx prisma migrate dev --name executive_report --create-only`. Then edit the generated migration.sql to add the partial unique index:
```sql
CREATE UNIQUE INDEX "ExecutiveReport_clientId_advisorProfileId_draft_unique"
    ON "ExecutiveReport"("clientId", "advisorProfileId")
    WHERE "status" = 'DRAFT';
```

Run `npx prisma generate` after migration creation.

Anti-patterns to avoid:
- Do NOT create a new enum for status -- reuse existing ReportStatus (per D-03)
- Do NOT add an assessmentId FK -- executive reports span multiple assessments; the snapshot records which assessmentIds were in scope
- Do NOT add a reportType discriminator column -- Executive Report and Advisor Brief are render-time variants of the same model, not separate DB rows (per D-19)

**File 1: src/lib/pdf/executive-report-types.ts**

Define the ExecutiveReportSnapshot interface following the schema in 25-RESEARCH.md Pattern 1. Key sub-interfaces:

- ExecutiveReportSnapshot: schemaVersion (1), reportingPeriod ({start, end, label}), clientName, generatedAt, pillarReadiness (PillarReadiness[]), executiveReadiness, scoreDelta (ScoreDeltaSummary | null), recommendationSummary (RecommendationSummary), engagementSummary (EngagementSummary | null), topPriorities (TopPriorityItem[]), intelligenceExcerpt (IntelligenceEvent[]), nextSteps (string[]), advisorNotes (string | null), meetingAgenda (string | null), discussionPrompts (string[]), assessmentIds (string[]), currentAssessmentId, previousAssessmentId (string | null)
- PillarReadiness: pillar, pillarLabel, score, riskLevel, impactLevel ("Critical"|"High"|"Medium"|"Low")
- ScoreDeltaSummary: deltas (PillarDelta[] -- import from reassessment-types.ts), overallDirection ("improved"|"regressed"|"mixed"|"unchanged"), keyDrivers (string[])
- RecommendationSummary: total, completed, inProgress, deferred, open, completionPct
- EngagementSummary: milestoneCompletionPct, totalMilestones, completedMilestones, overdueMilestones
- TopPriorityItem: name, category, impactLevel, status
- IntelligenceEvent: action, label, occurredAt (ISO string)

Export two pure derivation functions:

1. deriveExecutiveReadiness(pillars: PillarReadiness[]): { tier: "Developing"|"Mature"|"Advanced", highestRiskDomains: string[], strongestDomains: string[], strategicPriorities: string[] }
   - Algorithm per D-10: Any CRITICAL riskLevel or 2+ HIGH = "Developing". 60%+ LOW = "Advanced". Otherwise "Mature".
   - highestRiskDomains: pillars with riskLevel CRITICAL or HIGH, sorted by score ascending
   - strongestDomains: pillars with riskLevel LOW, sorted by score descending
   - strategicPriorities: derive from highest-risk domains ("Address [pillar] risk exposure")

2. deriveImpactLevel(urgencyScore: number, pillarWeight: number): "Critical"|"High"|"Medium"|"Low"
   - Algorithm per D-27: composite = urgencyScore * (pillarWeight / 16). >=7 Critical, >=5 High, >=3 Medium, else Low.

**File 2: src/lib/pdf/build-executive-report-snapshot.ts**

Add "server-only" import at top (same as score-delta.ts pattern).

Export buildExecutiveReportSnapshot(clientId: string, advisorProfileId: string, options?: { periodStart?: Date; periodEnd?: Date }): Promise<ExecutiveReportSnapshot>.

Implementation:
1. Determine reporting period: options.periodEnd ?? new Date() for end. For start: find last published ExecutiveReport for this client+advisor pair to get its reportingPeriodEnd as the new start. If none found, use earliest assessment.startedAt (D-22: first report covers all time).
2. Batch queries with Promise.all (avoid N+1 per Pitfall 1):
   - prisma.assessment.findMany for completed assessments within period, include: { scores: true }, orderBy: { completedAt: "asc" }
   - getGuidancePackageForClient(clientId, advisorProfileId) for recommendation counts
   - getEngagementClients(advisorProfileId) then filter for this clientId to get per-client engagement (per Open Question 3)
   - prisma.solutionActivity.findMany for intelligence timeline excerpt, where: { assessmentId: { in: assessmentIds } }, orderBy: { createdAt: "desc" }, take: 10
3. Determine current + previous assessment from the chain (latest by completedAt, follow previousAssessmentId).
4. If previous assessment exists: call computePillarDeltas with previous scores, current scores, and completed recommendations (filter GuidancePackage for COMPLETED status). Build ScoreDeltaSummary with overallDirection computed from per-pillar directions.
5. Build pillarReadiness array from current assessment scores. For each pillar, call deriveImpactLevel using the recommendation's urgencyScore and a PILLAR_WEIGHTS map (define at module top, values from existing assessment config -- if not readily available, use uniform weight of 8).
6. Call deriveExecutiveReadiness(pillarReadiness) for the executive readiness tier.
7. Build recommendationSummary from GuidancePackage counts.
8. Build engagementSummary from the filtered EngagementClientRow (completedCount, totalCount, completionPct), or null if no engagement data.
9. Build topPriorities: open/in-progress recommendations sorted by impactLevel descending, take top 5.
10. Build nextSteps: derive from top priorities and upcoming cadence ("Schedule reassessment by [date]" if ReviewCadence exists).
11. Format reportingPeriod.label using date-fns format (e.g., "January 1, 2026 -- June 27, 2026") per D-24.
12. Return the complete ExecutiveReportSnapshot with advisorNotes/meetingAgenda/discussionPrompts as null (these are populated from the ExecutiveReport row at render time, not during snapshot assembly).
