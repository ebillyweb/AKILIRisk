**src/lib/actions/executive-report-actions.ts**

"use server" at top. Follow report-actions.ts patterns exactly.

Import ReportActionResult type pattern. Create private helpers:
- requireSession(): same as analog
- authorizeForClient(session: SessionInfo, clientId: string): check isPlatformAdminRole first (ADMIN bucket). Then ADVISOR bucket: find advisorProfile by userId, find active ClientAdvisorAssignment. Return { ok: true, bucket, advisorProfileId } or { ok: false, code }. Different from analog's authorizeForAssessment because executive reports scope to client, not assessment.

Export these server actions:

1. generateExecutiveReport(input: { clientId: string; periodStart?: string; periodEnd?: string }): ReportActionResult<{ reportId: string }>
   - Zod-validate input (clientId required, dates optional ISO strings)
   - requireSession + authorizeForClient
   - Check for existing DRAFT: if one exists, return it (idempotent like getOrCreateDraft)
   - Determine reporting period: parse periodStart/periodEnd if provided (D-23). Default: periodEnd = now, periodStart = last published ExecutiveReport's reportingPeriodEnd for this client+advisor, or earliest assessment if first report (D-22)
   - Determine version: count existing ExecutiveReport rows for (clientId, advisorProfileId) + 1
   - prisma.executiveReport.create with status DRAFT, reportingPeriodStart/End, clientId, advisorProfileId, version
   - Return { ok: true, data: { reportId } }

2. saveExecutiveDraftEdits(input: { reportId: string; advisorNotes?: string; meetingAgenda?: string; discussionPrompts?: string[] }): ReportActionResult<{ saved: true }>
   - Zod-validate (advisorNotes max 2000 chars, meetingAgenda max 2000, discussionPrompts max 10 items each max 500 chars)
   - requireSession + fetch report + authorize (must be DRAFT, must be author's client)
   - prisma.executiveReport.update with provided fields
   - Return { ok: true, data: { saved: true } }. No audit row for editorial saves (per analog pattern).

3. publishExecutiveReport(reportId: string): ReportActionResult<{ publishedId: string; version: number }>
   - requireSession + fetch draft + authorize
   - If draft.status !== "DRAFT", return { ok: false, code: "not_draft" }
   - Build snapshot OUTSIDE transaction (expensive, read-only): buildExecutiveReportSnapshot(draft.clientId, draft.advisorProfileId, { periodStart: draft.reportingPeriodStart, periodEnd: draft.reportingPeriodEnd })
   - Build branding: buildBrandingSnapshot using latest completed assessment for this client
   - Overlay advisor fields: set snapshot.advisorNotes = draft.advisorNotes, snapshot.meetingAgenda = draft.meetingAgenda, snapshot.discussionPrompts = draft.discussionPrompts ?? []
   - prisma.$transaction:
     a. Re-read draft under txn (detect races -- if gone, throw "draft_disappeared")
     b. updateMany: set status SUPERSEDED for all PUBLISHED rows with same clientId + advisorProfileId
     c. update draft: status -> PUBLISHED, executiveSnapshotData -> snapshot JSON, brandingSnapshot -> branding JSON, publishedAt -> new Date(), publishedById -> session.userId
     d. Create new DRAFT at version+1 (next edit slot, inheriting advisorNotes/meetingAgenda/discussionPrompts)
   - void writeAudit after transaction: action "EXECUTIVE_REPORT_PUBLISH", entityType "ExecutiveReport". Add EXECUTIVE_REPORT_PUBLISH to AUDIT_ACTIONS constant if it uses a const object pattern, otherwise use string literal.
   - Catch P2002 -> { ok: false, code: "concurrent_publish", message: "Another publish landed first. Refresh and try again." }
   - Catch "draft_disappeared" -> { ok: false, code: "concurrent_publish" }

**src/lib/reports/executive-report-queries.ts**

No auth gating (caller's responsibility, matching analog comment pattern).

1. Export ExecutiveReportListRow interface: id, version, status ("DRAFT"|"PUBLISHED"|"SUPERSEDED"), reportingPeriodStart (Date), reportingPeriodEnd (Date), publishedAt (Date | null), publishedById (string | null), hasAdvisorNotes (boolean), createdAt (Date), updatedAt (Date).

2. getExecutiveReportListForClient(clientUserId: string, advisorProfileId: string): Promise<ExecutiveReportListRow[]>

   - Map to ExecutiveReportListRow, compute hasAdvisorNotes = !!row.advisorNotes

   - Single fetch for the edit form. Return null if not found.

**src/app/api/cron/executive-report-drafts/route.ts**

GET route following review-cadence/route.ts cron pattern exactly.

1. CRON_SECRET Bearer token validation with timingSafeEqual (copy pattern verbatim from review-cadence route)
2. Find ReviewCadence rows where nextDueDate is within the next 7 days (approaching) and the client has at least one COMPLETED assessment
3. For each cadence:
   a. Check for existing DRAFT ExecutiveReport for (cadence.clientId, cadence.advisorProfileId) -- skip if one exists (idempotent per Pitfall 5)
   b. Determine reporting period: periodEnd = now, periodStart = last published report's periodEnd or earliest assessment
   c. Determine version: count existing rows + 1
   d. Build snapshot: buildExecutiveReportSnapshot(cadence.clientId, cadence.advisorProfileId, { periodStart, periodEnd })
   e. prisma.executiveReport.create with status DRAFT, executiveSnapshotData: snapshot (pre-populate for advisor review), version, reporting period
4. Return JSON summary: { processed: N, draftsCreated: M, skipped: K }

Per D-21: scheduled reports are ALWAYS drafts, never auto-distributed. The advisor must explicitly publish.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Advisor -> Server Actions | Advisor must have active assignment to client |
| Cron -> Executive Report Creation | CRON_SECRET auth prevents external invocation |
| Advisor Brief -> Client Access | Clients must not access Advisor Brief content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-09 | Elevation of Privilege | generateExecutiveReport | mitigate | authorizeForClient checks active ClientAdvisorAssignment before creating DRAFT |
| T-25-10 | Elevation of Privilege | publishExecutiveReport | mitigate | Same auth check + DRAFT status guard. $transaction prevents race conditions. |
| T-25-11 | Spoofing | cron route | mitigate | CRON_SECRET Bearer token with timingSafeEqual (identical to review-cadence route) |
| T-25-12 | Denial of Service | cron route double-draft | mitigate | Check existing DRAFT before creating; partial unique index prevents DB-level duplicates |
| T-25-13 | Tampering | saveExecutiveDraftEdits | mitigate | Zod validation on all inputs with max lengths; only DRAFT rows editable |
| T-25-14 | Information Disclosure | Advisor Brief pages in UI | mitigate | Preview/download links for Advisor Brief only shown on advisor pages (never client-facing) |
</threat_model>
