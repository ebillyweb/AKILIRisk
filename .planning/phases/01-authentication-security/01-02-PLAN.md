  <action>
    1. Create `src/lib/email.ts`:
       - Initialize Resend client with RESEND_API_KEY
       - Export `sendPasswordResetEmail(email: string, resetUrl: string)` function
       - Email content: Subject "Reset your Belvedere password", HTML body with reset link, 15-minute expiry warning, "if you didn't request this" disclaimer
       - Use FROM_EMAIL env var (default to "onboarding@resend.dev" for development)
       - Handle errors gracefully -- log but don't expose email delivery failures to client (prevents enumeration)

    2. Create `src/app/api/auth/forgot-password/route.ts` (POST):
       - Validate email with Zod
       - Apply rate limiting: `resetLimiter` (3 requests per hour per email)
       - Look up user by email. If not found, return same success response (prevent email enumeration)
       - Generate reset token: `crypto.randomBytes(32).toString("hex")`
       - Hash token with SHA-256 before storing: `crypto.createHash("sha256").update(token).digest("hex")`
       - Store in VerificationToken table: identifier=email, token=hashedToken, expires=now+15min, used=false
       - Delete any existing unexpired tokens for this email first (prevent token spam)
       - Build reset URL: `${process.env.NEXT_PUBLIC_URL || "http://localhost:3000"}/reset-password?token=${rawToken}&email=${email}`
       - Send email via `sendPasswordResetEmail()`
       - Always return 200 with `{ message: "If an account exists with that email, a reset link has been sent" }`

    3. Create `src/app/api/auth/reset-password/route.ts` (POST):
       - Accept `{ token, email, password }` in request body
       - Validate password with same Zod schema as registration (min 8 chars, complexity requirements)
       - Hash the provided token with SHA-256
       - Look up in VerificationToken where: token=hashedToken, identifier=email, expires > now, used=false
       - If not found: return 400 `{ error: "Invalid or expired reset link" }`
       - Hash new password with Argon2id (same params as registration)
       - Update user password via `prisma.user.update()`
       - Mark token as used OR delete it (delete is cleaner): `prisma.verificationToken.delete()`
       - Invalidate all existing sessions for this user: `prisma.session.deleteMany({ where: { userId } })`
       - Return 200 `{ message: "Password reset successfully" }`

    4. Create `src/app/(auth)/forgot-password/page.tsx`:
       - Client component with email input form
       - On submit: POST to /api/auth/forgot-password
       - Show success state: "If an account exists with that email, we've sent a reset link. Check your inbox."
       - Show the success message regardless of API response (consistent UX, prevents enumeration)
       - Link back to "Return to sign in"

    5. Create `src/app/(auth)/reset-password/page.tsx`:
       - Client component that reads `token` and `email` from URL search params
       - If missing token/email, show error: "Invalid reset link"
       - Form: new password input, confirm password input, submit button
       - Client-side validation: passwords match, minimum 8 chars
       - On submit: POST to /api/auth/reset-password with { token, email, password }
       - On success: show "Password reset successfully" with link to sign in
       - On error: show error message (expired/invalid token)

    6. Update the sign-in page (`src/app/(auth)/signin/page.tsx`) to include a "Forgot password?" link pointing to /forgot-password (if not already added in Plan 01).
  </action>
