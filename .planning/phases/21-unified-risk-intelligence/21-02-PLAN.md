5. **Add advisor override test** — this exercises the `rulesOverride` path through the engine, proving that when an advisor supplies a filtered ruleset, only services from that subset are returned.

   **CRITICAL: Type shape conversion.** `PRODUCTION_CATALOG_RULES` is `CatalogRule[]` (fixture shape with `serviceRecommendationId`/`triggerConditions`), but `matchAndDedupeRecommendations` expects `RecommendationRule[]` (engine shape with `serviceId`/`conditions`). You MUST map the filtered rules to `RecommendationRule` shape before passing them as `rulesOverride`. The engine uses `rule.serviceId` to look up services via `getServiceRecommendation` — if `serviceId` is undefined (because you passed a `CatalogRule` with `serviceRecommendationId` instead), every lookup returns null and the test produces zero recommendations, causing a false pass on `expect(ids.every(...))` (vacuously true on empty array) before failing on the sorted-equality check.

   ```typescript
   import { RecommendationRule } from "./recommendation-engine";

   it("advisor rulesOverride restricts recommendations to overridden ruleset", async () => {
     const engine = new RecommendationEngine();
     // Filter to governance-only rules, then map CatalogRule -> RecommendationRule shape
     const governanceOnlyRules: RecommendationRule[] = PRODUCTION_CATALOG_RULES
       .filter(r => r.serviceRecommendationId.startsWith("governance_"))
       .map(r => ({
         id: r.id,
         serviceId: r.serviceRecommendationId,       // CatalogRule field -> engine field
         conditions: r.triggerConditions,              // CatalogRule field -> engine field
         priority: r.priority,
       }));
     // Feed high-risk answers and scores for ALL pillars, but override rules to governance-only
     const recs = await engine.matchAndDedupeRecommendations(
       {
         assessmentId: "as-override",
         userId: "u-override",
         pillarScores: HIGH_RISK_PILLAR_SCORES,
         answers: HIGH_RISK_FAMILY_ANSWERS,
         householdProfile: null,
         missingControls: [],
       },
       governanceOnlyRules,   // <-- rulesOverride parameter (RecommendationRule[], NOT CatalogRule[])
     );
     const ids = recs.map(r => r.id);
     // Only governance services should appear despite all pillars being high-risk
     expect(ids.sort()).toEqual([...GOVERNANCE_REMEDIATION_SERVICE_IDS].sort());
     // No non-governance services should leak through
     expect(ids.every(id => id.startsWith("governance_"))).toBe(true);
   });
   ```

   The `toPrismaCatalogMocks` helper at the top of the test file performs a similar mapping for the `findMany` mock (Prisma row shape), but for `rulesOverride` the engine expects `RecommendationRule` directly — not Prisma rows. The mapping in `loadRecommendationRules` (line ~276 of recommendation-engine.ts) shows the canonical conversion: `serviceRecommendationId -> serviceId`, `triggerConditions -> conditions`.
